# WorkOutApp プロジェクト固有ルール

## 問題対応フロー（必須）

問題やエラーが発生した際は、以下のフローを必ず遵守すること：

### 1. 問題発見時
- **修正作業を開始する前に、必ず problem.txt に記録する**
- 記録なしで修正作業を開始することは禁止

### 2. 原因調査
- 問題の根本原因を徹底的に調査する
- 表面的な対処ではなく、なぜ発生したかを特定する

### 3. 修正計画の作成
- 調査結果に基づき、具体的な修正計画を立てる
- 修正の優先順位を明確にする

### 4. problem.txt への記録
以下のフォーマットで記録：
```
## Problem #[番号]: [問題の簡潔なタイトル]
発生日: YYYY-MM-DD
関連Phase/Work: Phase X / Work X
ステータス: 調査中 / 修正中 / 解決済み / 保留

### 問題の詳細
[何が起きたか、エラーメッセージ、再現手順など]

### 原因分析
[なぜ起きたか、根本原因]

### 試したアプローチ（失敗したものも含む）
1. [アプローチ1] → 結果: [成功/失敗/理由]

### 採用した解決策
[最終的に採用した解決方法]

### 再発防止策
[同じ問題を起こさないための対策]

### 学んだこと
[今後に活かすべき教訓]

### 解決日
YYYY-MM-DD
```

### 5. 修正実行
- 計画に基づいて修正を実施
- 修正完了後、結果を problem.txt に追記

### 6. 類似問題の参照
- 同じ/類似の問題が発生した場合、過去の記録を必ず参照してから対応

## Phase完了時のチェック

各Phase完了時に必ず実施：
1. `npm run build` でビルドが通ることを確認
2. `npx tsc --noEmit` で型チェックが通ることを確認
3. エラーがある場合は上記の問題対応フローを実行

## ルート定義のルール

- ページが実装されていない状態でルート定義を追加しない
- ルート追加時は対応するページ実装を同時に行う
