# WorkOutApp プロジェクト固有ルール

## ⚠️ 最重要ルール 1: タスク管理の徹底（必須・厳守）

**全てのタスクは progress.txt と manual-work.txt で管理すること。**
このルールは絶対に省略してはならない。

### タスク管理ファイル
| ファイル | 用途 |
|---------|------|
| `progress.txt` | AI作業タスク（コード実装、修正、テスト等） |
| `manual-work.txt` | 手動作業タスク（Xcode設定、App Store設定、証明書等） |
| `problem.txt` | 問題・エラーの記録と解決策 |

### 必須ルール
1. **新規タスク着手時**: 必ず progress.txt にPhaseとして追加してから作業開始
2. **タスク完了時**: 必ず progress.txt に [完了: YYYY-MM-DD] を追記
3. **手動作業タスク**: manual-work.txt に追加し、ステータスを管理
4. **問題発生時**: problem.txt に記録してから修正開始
5. **Phase完了時**: 全タスク完了確認後、archive.txt に移動

### 違反時の対応
- タスクを progress.txt に記録せずに作業を開始した場合、即座に作業を中断
- まず progress.txt にタスクを追加してから作業を再開する

## ⚠️ 最重要ルール 2: 問題対応フロー（必須・厳守）

**このルールは絶対に省略してはならない。**
問題やエラーが発生した際は、以下のフローを**必ず順番に**遵守すること。

### フロー概要

```
問題発生 → problem.txt確認 → TodoWrite追加 → 原因調査 → 修正計画 → problem.txt記録 → 修正実施 → 動作確認 → TodoWrite完了 → problem.txt更新
```

### 0. problem.txt の過去事例を確認（最初に必ず実行）

**修正作業を開始する前に**、まず problem.txt を読んで：
- 類似の問題が過去に発生していないか確認
- 過去の解決策や学んだことを参照
- 同じ失敗を繰り返さないようにする

### 1. TodoWrite にタスクを追加

problem.txt 確認後、**修正作業を開始する前に**以下のタスクを TodoWrite に追加：

```
- [ ] Problem #X: 原因調査
- [ ] Problem #X: 修正計画作成
- [ ] Problem #X: problem.txt に記録（修正前）
- [ ] Problem #X: 修正実施
- [ ] Problem #X: 動作確認
- [ ] Problem #X: problem.txt 更新（修正後）
```

### 2. 原因調査
- 問題の根本原因を徹底的に調査する
- 表面的な対処ではなく、なぜ発生したかを特定する
- エラーメッセージ、スタックトレース、関連ファイルを確認

### 3. 修正計画の作成
- 調査結果に基づき、具体的な修正計画を立てる
- 複数のアプローチがある場合は比較検討する
- 修正するファイルと変更内容を明確にする

### 4. problem.txt への記録（修正前）
- problem.txt のテンプレートに従って記録
- **必須チェックリスト**を含めること
- **記録なしで修正作業を開始することは禁止**

### 5. 修正実行
- 計画に基づいて修正を実施
- 失敗したアプローチも記録する

### 6. 動作確認
- 問題が解決したことを確認
- 副作用がないことを確認
- ビルドが通ることを確認：
  - iOS: `xcodebuild` または Xcode でビルド
  - TypeScript: `npx tsc --noEmit`
  - Web: `npm run build`

### 7. TodoWrite タスク完了 & problem.txt 更新
- 全てのタスクを完了にする
- problem.txt のステータスを「解決済み」に変更
- 必須チェックリストを全てチェック
- 「採用した解決策」「学んだこと」「再発防止策」を記入

### 違反時の対応
- **このフローを省略して修正を開始した場合、即座に作業を中断**
- 最初からフローをやり直す
- 中途半端な修正は revert する

## problem.txt 記録テンプレート

新しい問題を記録する際は、以下のテンプレートを使用すること：

```markdown
## Problem #[番号]: [問題の簡潔なタイトル]
発生日: YYYY-MM-DD
関連Phase/Work: Phase X / Work X
ステータス: 調査中 / 修正中 / 解決済み / 保留

### 必須チェックリスト（全て完了必須）
- [ ] 1. problem.txt の過去事例を確認した
- [ ] 2. TodoWrite に問題対応タスクを追加した
- [ ] 3. 原因を徹底調査した
- [ ] 4. 修正計画を作成した
- [ ] 5. problem.txt に記録した（この項目）
- [ ] 6. 修正を実施した
- [ ] 7. 動作確認を完了した
- [ ] 8. TodoWrite のタスクを完了にした

### 問題の詳細
[何が起きたか、エラーメッセージ、再現手順など]

### 原因分析
[なぜ起きたか、根本原因]

### 修正計画
[具体的な修正手順をステップで記載]

### 修正したファイル
[修正対象ファイルリスト]

### 試したアプローチ（失敗したものも含む）
1. [アプローチ1] → 結果: [成功/失敗/理由]

### 採用した解決策
[最終的に採用した解決方法]

### 動作確認結果
- [ ] 問題が解決したことを確認
- [ ] 副作用がないことを確認
- [ ] ビルドが通ることを確認

### 再発防止策
[同じ問題を起こさないための対策]

### 学んだこと
[今後に活かすべき教訓]

### 解決日
YYYY-MM-DD
```

## Phase完了時のチェック

各Phase完了時に必ず実施：
1. `npx tsc --noEmit` で型チェックが通ることを確認
2. iOS: `xcodebuild` でビルドが通ることを確認
3. エラーがある場合は上記の問題対応フローを実行

## ⚠️ 「アップデート」隠語ルール（必須・厳守）

**このアプリのUI上では「メッセージ」という単語を絶対に使用しない。**
代わりに「アップデート」という隠語を使用すること。

### 適用範囲
- ✅ UIテキスト（画面に表示されるすべての文字列）
- ✅ 翻訳ファイル（ja.json, en.json）
- ❌ コード内の変数名・関数名・コメント（対象外）

### 変換例
| ❌ 使用禁止 | ✅ 使用する |
|------------|-----------|
| 新しいメッセージ | 新しいアップデート |
| メッセージが届きました | アップデートがあります |
| 1件のメッセージ | 1件のアップデート |
| メッセージを送信 | （使用しない - 内部機能として隠す）|

### 通知テキスト
- タイトル: 「Health Manager」
- 本文: 「新しいアップデートが○件あります」

### 理由
このアプリは「ヘルスケアアプリ」として偽装されており、
ユーザーのプライバシーを保護するため、外部から見てメッセージアプリと
わからないようにする必要がある。

## ルート定義のルール

- ページが実装されていない状態でルート定義を追加しない
- ルート追加時は対応するページ実装を同時に行う

## React Native + Firebase の注意事項（Problem #2 より）

### 互換性確認
- React Native バージョンアップ時は Firebase との互換性を事前確認
- react-native-firebase の GitHub Issues をチェック

### ネイティブ依存の最小化
- 可能な限り Firebase JS SDK を使用
- ネイティブモジュールが必要な場合は互換性を十分検証

### プッシュ通知
- Firebase JS SDK は FCM をサポートしない
- 代替: Expo Push Notifications, OneSignal

---

## 過去の問題からの教訓

### Problem #1: Webアプリビルドエラー
- ルート定義と実装は必ずセット
- 型定義は最初に統一
- Phase完了時にビルドチェック

### Problem #2: iOS Xcodeビルドエラー（Firebase互換性）
- React Native アップグレード時はネイティブモジュールとの互換性を必ず事前確認
- 複数アプローチ失敗時は根本的なアーキテクチャ変更を検討
- Podfile設定調整には限界がある場合がある
