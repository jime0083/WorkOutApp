## Problem #1: Webアプリビルドエラー（複数の型エラー・未実装モジュール）
発生日: 2026-05-02
関連Phase/Work: Phase 8.5（国際化）実装中に発見
ステータス: 解決済み

### 問題の詳細
Phase 8.5の国際化実装後、`npm run build` を実行したところ、30以上のTypeScriptエラーが発生しビルドが失敗した。

**エラー一覧:**
```
1. 存在しないモジュールのインポート (4件)
   - src/routes/index.tsx: ProfilePage, SubscriptionPage, SettingsPage, AuthCallbackPage

2. Firebase Functions エクスポート漏れ (3件)
   - useMessageLimit.ts, friend.ts, message.ts: 'functions' が firebase.ts からエクスポートされていない

3. LoadingSize 型の不一致 (7件)
   - 複数のファイルで "large", "small" を使用しているが、型定義は 'sm' | 'md' | 'lg'

4. FullScreenLoading の visible プロパティ欠落 (1件)
   - routes/index.tsx: <FullScreenLoading /> に visible が渡されていない

5. verbatimModuleSyntax 関連 (4件)
   - auth.ts, authStore.ts: 型のみのインポートに `import type` が必要

6. AuthState エクスポート問題 (1件)
   - stores/index.ts: AuthState が authStore.ts からエクスポートされていない

7. 未使用変数 (6件)
   - MediaPreview.tsx: useCallback
   - AddFriendPage.tsx, ProfileEditPage.tsx: navigate
   - message.ts: Timestamp, QueryDocumentSnapshot, CreateMessageInput, Conversation

8. Date/Timestamp 変換問題 (4件)
   - ConversationItem.tsx, MessageBubble.tsx, MessageList.tsx, MessageSearch.tsx

9. uiStore.ts の型問題 (1件)
   - null が string | undefined に割り当てられない
```

### 原因分析

#### 根本原因1: 先行実装の問題
routes/index.tsx で Phase 9, 11 で実装予定のページ（SubscriptionPage, SettingsPage等）を先にインポートしていた。
→ ルート定義を先行して作成したが、対応するページ実装が追いついていない

#### 根本原因2: Firebase Functions 設定漏れ
firebase.ts で Cloud Functions のインスタンス（getFunctions）を初期化・エクスポートしていない。
→ Phase 7-8 の実装時に Functions を使用するコードを書いたが、firebase.ts の更新を忘れた

#### 根本原因3: 型定義の不整合
Loading コンポーネントの型定義（'sm' | 'md' | 'lg'）と使用側（'small' | 'large'）が一致していない。
→ コンポーネント作成時と使用時で命名規則が統一されていなかった

#### 根本原因4: TypeScript 設定の厳格化
verbatimModuleSyntax が有効な状態で、型インポートの書式が厳格に要求されている。
→ TypeScript 6.0 の厳格な設定に対応していないコードが残っていた

#### 根本原因5: 開発中の未使用コード
開発途中で使用予定だったインポートや変数が残っている。
→ 開発フェーズ間の整理が不十分

- **推奨**: 使用側を 'sm' | 'md' | 'lg' に統一（既存の型定義を維持）

```typescript
// 選択肢2: Suspense用のラッパーコンポーネントを作成
```

- **推奨**: 選択肢1（シンプル）

**A-5. verbatimModuleSyntax 対応**

**A-6. AuthState エクスポート**

**A-7. 未使用変数の削除**
- 各ファイルから未使用のインポートと変数を削除

**A-8. Date/Timestamp 変換の修正**

### 学んだこと
1. **先行実装の危険性**: 未実装のページをルートに追加すると、ビルド時にエラーになる。必ず実装とルート定義はセットで行う。
2. **型定義の統一の重要性**: コンポーネント作成時に型定義を決めたら、使用側も同じ規則に従う必要がある。
3. **verbatimModuleSyntax**: TypeScript 6.0 以降は型のみのインポートに `import type` を使用する必要がある。
4. **Firestore Timestamp**: Date と Timestamp の両方に対応するには、`instanceof Date` チェックと `unknown` を経由した型アサーションが有効。
5. **Phase完了時のビルドチェック**: 各Phase完了時に必ず `npm run build` を実行することで、問題の早期発見が可能。

発生日: 2026-05-09
関連Phase/Work: Phase 13（ビルド・デプロイ準備）
ステータス: 解決済み

### 必須チェックリスト（全て完了必須）

- [x] 2. 原因を徹底調査した

- [x] 4. problem.txt に記録した（この項目）
- [x] 5. 修正を実施した
- [x] 6. 動作確認を完了した
- [x] 7. TodoWrite のタスクを完了にした

### 問題の詳細

iOS Xcodeビルドを実行した際、複数のエラーが連続して発生しビルドが失敗した。

**主要なエラー:**
```
1. gRPC-Core.modulemap not found
   - module map file '/Users/.../Pods/Headers/Private/grpc/gRPC-Core.modulemap' not found

2. FirebaseAuth-Swift.h not found
   - 'FirebaseAuth-Swift.h' file not found

3. RCTPromiseRejectBlock not found
   - Unknown type name 'RCTPromiseRejectBlock'
   - Unknown type name 'RCTBridgeModule'

4. Non-modular header includes
   - Include of non-modular header inside framework module
```

### 原因分析

#### 根本原因: @react-native-firebase と React Native 0.85.2 の互換性問題

1. **gRPC依存性の問題**
   - `@react-native-firebase/firestore` は `gRPC` をネイティブ依存として使用
   - `use_modular_headers!` グローバル設定がgRPCのmodulemap参照に問題を起こす
   - React Native 0.85 の新アーキテクチャ（New Architecture）とgRPCの構造が衝突

2. **Swiftブリッジングヘッダーの問題**
   - Firebase SDK v23+ はSwiftモジュールを含む
   - 静的ライブラリ (`use_frameworks!` なし) ではSwiftブリッジングヘッダーが生成されない
   - `FirebaseAuth-Swift.h` が見つからないエラーが発生

3. **React Native Objective-Cモジュールの問題**
   - `use_frameworks! :linkage => :dynamic` を使用するとフレームワーク間のインポート順序に問題
   - `RCTPromiseRejectBlock` などのReact Nativeマクロが未定義エラー

4. **技術的詳細**
   - React Native 0.85 は Hermes エンジンをデフォルトで使用
   - プリビルドされた React Native Core がモジュール構造を厳格に要求
   - Firebase iOS SDK の BoringSSL/gRPC 依存が複雑な Pod 構成を必要とする

### 修正計画

#### 検討した選択肢

| 選択肢 | 内容 | 評価 |
|--------|------|------|
| A | Podfile設定を調整（modular_headers, use_frameworks） | ❌ 複数のアプローチを試したが全て失敗 |
| B | React Native をダウングレード（0.73等） | ❌ 大規模な変更が必要、他の依存関係に影響 |
| C | Firebase JS SDK に移行 | ✅ 採用：ネイティブ依存を排除 |

#### 採用した計画: Firebase JS SDK への移行

```
Phase 1: パッケージ変更
  - @react-native-firebase/* パッケージを削除
  - firebase (JS SDK) パッケージをインストール

Phase 2: Podfile更新
  - Firebaseネイティブ依存を削除
  - シンプルなPodfile構成に変更

Phase 3: サービスコード更新
  - firebase.ts: JS SDK初期化に書き換え
  - auth.ts: JS SDK API使用に更新
  - subscription.ts: Firestore JS SDK + react-native-iap v15対応
  - notification.ts: スタブ実装（FCMはJS SDKでサポートされない）
  - messages.ts: Firebase Functions JS SDK使用
  - 各スクリーンファイル: Firestoreクエリを更新

Phase 4: ビルド確認
  - pod install
  - xcodebuild
```

### 試したアプローチ（失敗したものも含む）

1. **use_modular_headers! を削除し個別設定** → 失敗
   - gRPCのmodulemap問題は解決したが、Swiftブリッジングヘッダー問題が発生

2. **use_frameworks! :linkage => :static 使用** → 失敗
   - "Swift Pods cannot be integrated as static libraries" エラー

3. **use_frameworks! :linkage => :dynamic 使用** → 失敗
   - RCTPromiseRejectBlock not found エラー

4. **Firebase SDK v23 → v22 ダウングレード** → 失敗
   - 同様の互換性問題が発生

5. **@react-native-firebase/firestore のみ削除** → 失敗
   - gRPCは削除されたが、他のFirebase SDK問題が残存

6. **Firebase JS SDK への移行** → 成功
   - ネイティブ依存を完全に排除し、ビルド成功

### 採用した解決策

**Firebase JS SDK への完全移行**

```typescript
// Before: @react-native-firebase
import auth from '@react-native-firebase/auth';
import firestore from '@react-native-firebase/firestore';

// After: Firebase JS SDK
import { getAuth, signInWithEmailAndPassword } from 'firebase/auth';
import { collection, doc, query, onSnapshot } from 'firebase/firestore';
import { getFirestoreInstance } from './firebase';
```

**主要な変更点:**
1. ネイティブモジュール依存を排除
2. Firebase初期化を単一のfirebase.tsで管理
3. Firestoreクエリパターンの変更（メソッドチェーン → 関数呼び出し）
4. FCM（プッシュ通知）はスタブ実装（JS SDKでサポートされないため）

   - 0.85 の New Architecture は多くのネイティブモジュールと互換性問題を起こす可能性がある
   - 特にgRPC、BoringSSL等の複雑な依存を持つライブラリは注意が必要

2. **Firebase ネイティブ vs JS SDK のトレードオフ**
   - ネイティブ SDK: 高性能、オフラインサポート、プッシュ通知対応
   - JS SDK: 互換性が高い、ネイティブ依存なし、一部機能制限あり

3. **Podfile設定の複雑さ**
   - `use_frameworks!`, `use_modular_headers!`, `linkage` の組み合わせは非常に複雑
   - 一つの問題を解決すると別の問題が発生することがある

4. **問題解決のアプローチ**
   - 複数のアプローチを試しても解決しない場合は、根本的なアーキテクチャ変更を検討
   - 時間をかけて小さな修正を繰り返すより、抜本的な解決策の方が効率的な場合がある

5. **事前調査の重要性**
   - 大規模なバージョンアップ（特にReact Native）前には互換性調査が必須
   - GitHub Issues、公式ドキュメント、コミュニティの情報を事前にチェック


### 原因分析

`firestore.rules` の `users` コレクションで `allow create: if false;` が設定されているため、クライアントからのドキュメント作成がすべてブロックされている。

```javascript

// 作成: Cloud Functions経由のみ（認証トリガーで作成）

```

元の設計ではCloud Function `onUserCreate` で自動作成する想定だったが:
1. 新しいデータモデルでは `password2Hash` が必要
2. Firebase Auth triggerではパスワードにアクセスできない
3. したがってクライアント側でドキュメントを作成する必要がある

### 修正計画

1. `firestore.rules` を変更:
   ```javascript
   // 変更前
   allow create: if false;

   // 変更後
   allow create: if isOwner(userId);
   ```

2. Firebase CLIでルールをデプロイ:
   ```bash
   firebase deploy --only firestore:rules
   ```

### 修正したファイル
- `firestore.rules`

### 試したアプローチ（失敗したものも含む）

2. firestore.rulesを変更してクライアントからの作成を許可 → 採用

### 採用した解決策
`firestore.rules` の `users` コレクションルールを変更:
```javascript
// 変更前
allow create: if false;

// 変更後
allow create: if isOwner(userId);
```

Firebase CLIでデプロイ:

### 問題の詳細

サブスク課金ボタンをタップした際、以下の表示が出た:
```
subscription.purchaseStartedTitle
subscription.purchaseStartedMessage
```

翻訳キーがそのまま表示されており、翻訳が見つかっていない。

### 原因分析

1. `subscription.purchaseStartedTitle` と `subscription.purchaseStartedMessage` が翻訳ファイル（ja.json, en.json）に存在しない
2. i18nextは存在しないキーをそのまま文字列として返すため、フォールバックの `||` が機能しない

### 修正計画

#### Phase 7-A: 翻訳キー追加
1. ja.json に `purchaseStartedTitle`, `purchaseStartedMessage` を追加
2. en.json に同様のキーを追加

#### Phase 7-B: UI変更（ユーザー要件）
1. 「プレミアムプラン」→「プランの選択」に変更
2. 3つのプランを表示:
   - フリープラン（月10通までのメッセージ送信）
   - 月額プラン（メッセージ無制限、画像・動画送受信、緊急全データ削除）
   - 年額プラン（月額の全機能を20%OFFで利用）

### 修正したファイル
- `mobile/src/i18n/locales/ja.json` - 翻訳キー追加、新しいプラン表示用テキスト追加
- `mobile/src/i18n/locales/en.json` - 同様の翻訳キー追加
- `mobile/src/screens/subscription/SubscriptionScreen.tsx` - 3プラン表示UIに完全書き換え
